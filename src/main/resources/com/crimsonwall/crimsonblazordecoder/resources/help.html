Crimson Blazor Decoder

Overview

Crimson Blazor Decoder is a ZAP add-on that decodes and displays Blazor Pack messages sent over WebSocket connections in real time.

Blazor Server applications communicate between the browser and server using the Blazor Pack protocol over SignalR WebSockets. The messages are encoded in a binary MessagePack format that is not human-readable. This makes security testing of Blazor Server applications difficult, as the WebSocket traffic in ZAP appears as opaque binary data.

Crimson Blazor Decoder intercepts these WebSocket messages, decodes the MessagePack payload, and presents the data as pretty-printed JSON in a dedicated ZAP panel, enabling security testers to inspect and analyse Blazor Pack traffic.

Why Use This Add-on?

Blazor Server is increasingly used in enterprise web applications. During a penetration test, being able to read Blazor Pack traffic is essential for:

  • Understanding application logic — Blazor Server renders UI server-side and pushes diffs to the client as RENDER_BATCH messages. Decoding these reveals which components are rendering, what data they contain, and how the UI state changes in response to user actions.
  • Identifying sensitive data exposure — Component state, form values, and server responses all travel over the WebSocket. Decoded messages make it straightforward to spot PII, tokens, or business logic that should not be visible to the client.
  • Mapping JS interop calls — JS_INTEROP messages show every JavaScript function invoked by the server, including method names and arguments. This is useful for finding dangerous sinks or undocumented client-side behaviour.
  • Identifying SignalR circuit endpoints — Circuit Start and Close messages reveal connection identifiers and negotiation details that can be used to test session handling and connection hijacking scenarios.

How It Works

  1. Intercepts WebSocket message frames (both text and binary) through a ZAP WebSocket observer.
  2. Detects Blazor/SignalR messages using protocol-specific heuristics (binary markers, JSON markers).
  3. Decodes the MessagePack payload, handling the Blazor Pack multi-value encoding, prefix bytes, and SignalR hub message format.
  4. Displays the decoded data in a syntax-highlighted JSON view with a hex dump of the raw bytes.
  5. Categorises messages by type: Render Batch, JS Interop, Circuit Start/Close, Completion, and more.

UI Panel

The main UI panel displays decoded Blazor Pack messages in a table with the following columns:

  • Time — Timestamp of when the message was captured
  • Type — Message type (Render Batch, JS Interop, Circuit Start, Circuit Close, Error, Unknown)
  • Direction — Client → Server (C→S) or Server → Client (S→C)
  • Summary — Brief description of the message content

Table Features

  • Auto-scroll — Automatically scrolls to show new messages. Click any row other than the last one to disable auto-scroll and manually inspect messages.
  • Color coding — Rows are color-coded by message type:
    • Light blue — Render Batch messages
    • Light green — JS Interop messages
    • Light red — Error messages
    • White — All other message types
  • Marking — Right-click any row to mark/unmark it for later reference. Marked rows are highlighted in lime green.
  • Regex highlighting — Rows containing regex matches are highlighted in yellow (highest priority color). Hover over highlighted rows to see which regex rules matched.
  • Message limit — Maximum of 10,000 messages kept in memory. Oldest messages are automatically removed when the limit is reached.

Detail View Tabs

When you select a message in the table, detailed information is shown in the tabbed view on the right:

JSON Tab

Displays the pretty-printed JSON representation of the decoded message with syntax highlighting:

  • Keys — Red
  • Strings — Green
  • Numbers — Orange
  • Booleans/null — Purple
  • Punctuation — Gray

Regex highlighting — If the message contains regex matches, matching text in the data section is highlighted with a yellow background. Only matches within the decoded data fields are highlighted — metadata fields (timestamp, messageId, rawPayload) are excluded to prevent false positives.

Copy — Right-click to copy selected text to the clipboard.

Raw Tab

Displays a hex dump of the original binary message payload with:

  • Offset column — Byte offset in hexadecimal (gray)
  • Hex column — Bytes in hexadecimal (green)
  • ASCII column — ASCII representation (blue)

Large payloads are truncated to 4096 bytes for display. The actual byte count is shown at the bottom.

Copy — Right-click to copy selected text to the clipboard.

Modify Tab (Outgoing Messages Only)

This tab appears only for outgoing (Client → Server) messages when the WebSocket connection is active. It allows you to:

  1. Edit the JSON payload in the text area
  2. Click Send to Server to resend the modified message

The send button is only enabled when the WebSocket connection is active. A status message below the button shows the result of the send operation (success, error, connection not available).

Use this feature to test server-side validation, input sanitisation, and business logic bypasses.

RegEx Tab

Shows which regex rules matched in the selected message. This tab displays the list of matching rule names. The matching text is also highlighted in yellow in the JSON view.

Regex Security Scanning

The Crimson Blazor Decoder includes a powerful regex-based security scanner that automatically highlights sensitive data patterns in decoded Blazor Pack messages.

Default Regex Rules

The add-on includes 25 pre-configured regex rules for detecting common sensitive data patterns:

  • Email addresses
  • IPv4 addresses
  • South African ID numbers
  • Credit card numbers (multiple formats)
  • AWS Access Keys
  • AWS Secret Keys
  • GCP Service Account credentials
  • GitHub personal access tokens
  • GitLab personal access tokens
  • JWT tokens
  • Private keys (RSA, DSA, EC, ECDSA, OPENSSH)
  • Stripe API keys
  • Slack tokens
  • Discord tokens
  • SendGrid API keys
  • Twilio API keys
  • Azure Storage account keys
  • Generic secret keys (key, token, secret, password, api_key)

How Regex Matching Works

  • Scope — Regex matching is scoped to the decoded data fields only. Metadata fields (timestamp, messageId, rawPayload) are excluded to prevent false positives.
  • Truncation — Payloads are truncated to 5000 characters for regex matching to maintain performance.
  • Safeguards — Protection against catastrophic backtracking: character access is limited to 100,000 per pattern match.
  • Direction filtering — Rules can be enabled separately for Client→Server and Server→Client messages.

Visual Indicators

  • Table rows — Yellow background highlight (overrides all other row colors)
  • JSON view — Yellow background on matching text within the data section
  • Tooltips — Hover over highlighted table rows to see which regex rules matched

Configuration

Access the add-on configuration via Tools → Options → Crimson Blazor Decoder.

Regex Rules Table

The options panel displays a table of all regex rules with the following columns:

  • Name — Descriptive name for the rule (editable)
  • Regex — Regular expression pattern (editable)
  • C→S — Enable for Client → Server messages
  • S→C — Enable for Server → Client messages

Managing Rules

  • Add — Click the Add button to create a new regex rule. Enter a name, pattern, and check the desired direction checkboxes.
  • Remove — Select a rule and click Remove to delete it.
  • Edit — Click directly on any cell to edit it. Pattern changes are validated for syntax errors.
  • Toggle all — Click the C→S or S→C column header to toggle all checkboxes in that column.

Rule Limits

  • Maximum of 200 regex rules allowed
  • Patterns are validated for syntax errors when added or edited
  • Invalid patterns are rejected with an error message

Configuration Persistence

Regex rules are automatically saved to ZAP's configuration file when you click OK in the Options dialog. Changes take effect immediately for new messages.

Toolbar Actions

Clear Messages

Click the Clear Messages button to remove all decoded messages from the table. A confirmation dialog is shown before clearing.

Export

Click the Export button to export the currently selected message:

  • If JSON tab is active — Exports as a .json file with the pretty-printed JSON representation
  • If Raw tab is active — Exports as a .raw file with the original binary payload

A file save dialog is shown with a default filename based on the message ID and timestamp. You will be prompted to confirm if the file already exists.

Requirements

  • ZAP 2.17.0 or later
  • The WebSocket add-on (installed by default in ZAP)

Accessing the Panel

After installation, open the panel via View → Show Tab → Crimson Blazor Decoder Tab.

Tips and Tricks

  • Disable auto-scroll — Click any row other than the last one to stop auto-scroll and manually inspect previous messages.
  • Mark important messages — Right-click and select Mark to highlight important messages in lime green.
  • Quick regex toggle — Click the C→S or S→C column header in the options table to quickly enable/disable all rules for that direction.
  • Timestamp tooltips — Hover over timestamp fields in the JSON view to see human-readable dates.
  • Modify and resend — Use the Modify tab on outgoing messages to test how the server handles modified payloads.
  • Hex dump analysis — Use the Raw tab to inspect the original binary format and understand the Blazor Pack encoding.

Performance Considerations

  • Maximum 10,000 messages stored in memory (oldest removed when limit reached)
  • JSON display truncated to 50,000 characters
  • Hex dump display truncated to 4,096 bytes
  • Regex matching limited to 5,000 characters per payload
  • Character access limited to 100,000 per regex pattern (prevents catastrophic backtracking)

Troubleshooting

No messages appearing?

  • Verify that the target application is using Blazor Server
  • Check that WebSocket connections are being proxied through ZAP
  • Ensure the WebSocket add-on is enabled

Messages not decoding correctly?

  • Some messages may not be Blazor Pack messages — only Blazor/SignalR messages are decoded
  • Binary frames without the correct Blazer Pack signature will be skipped
  • Check the Raw tab to see the original binary payload

Regex matches not appearing?

  • Verify that the regex rules are enabled for the correct direction (C→S or S→C)
  • Check that the pattern syntax is valid (invalid patterns are rejected)
  • Regex matching is limited to the decoded data fields — metadata fields are excluded
